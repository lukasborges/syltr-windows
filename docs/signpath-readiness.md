# SignPath Foundation readiness

This checklist separates repository preparation from actions that require a
SignPath account or approval.

Application status: submitted on 16 July 2026 and awaiting SignPath Foundation
review. No signing identifiers or certificate subject have been assigned yet.

## Prepared in the repository

- public source repository: `lukasborges/syltr-windows`;
- OSI-approved `GPL-3.0-or-later` license;
- application behavior and build instructions in the README;
- privacy, security, uninstallation and code-signing policies;
- signing candidates configured to be generated by a GitHub-hosted Windows
  runner;
- unsigned workflow artifact uploaded before any signing request;
- public release permission withheld from the unsigned candidate workflow.

## External application steps

1. Confirm multi-factor authentication is enabled for the GitHub maintainer.
2. Submit the application at <https://signpath.org/apply> and accept the
   SignPath Foundation conditions. Completed on 16 July 2026.
3. Ask SignPath whether the current pre-beta state satisfies its requirement
   that the project already be released. Do not publish an unsigned MSIX merely
   to satisfy that condition.
4. After acceptance, enable multi-factor authentication in SignPath and install
   the SignPath GitHub App for this repository.
5. Record the organization ID, project slug, signing policy slug and artifact
   configuration slug assigned by SignPath.
6. Create the `SIGNPATH_API_TOKEN` repository secret with submitter-only
   permissions.
7. Replace the provisional MSIX publisher with the exact certificate subject
   supplied by SignPath.

## Workflow activation after approval

The signed workflow will upload the unsigned MSIX with
`actions/upload-artifact`, pass that artifact ID to
`signpath/github-action-submit-signing-request@v2`, wait for manual approval
and download the signed result. It must then validate the signature and
publisher before generating the App Installer descriptor or creating a GitHub
Release.

No placeholder signing step is kept in the active workflow: a partially
configured action would either fail every candidate build or weaken the
current explicit publication gate.
